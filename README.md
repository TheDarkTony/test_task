Технологии: DRF + Postgres
Подход: database first


Приложение представляет собой управление тренировками (дневник) в фитнес клубе.


В клубе есть роли админа, тренера и спортсмена.
За управление доступа к ресурсам приложения ответственна сущность permissions (c таким же именем таблицы в базе данных)


permissions хранит следующие свойста:
- id (pk рекорда)
- entity (ресурс, к которому применяется разрешения)
- role (роль, к которая наделяется разрешения)
- mode (хранит два значения: self и other)
- allow_create (контролирует ограничение на создание новых "records")
- allow_edit (контролирует ограничение на редактирование "records")
- allow_read (контролирует ограничение на чтение "records")
- person_id (если не пустое, то раззрешение преобладает над теми разрешениями, которые в одной entity,role,mode. персональное ограничение участника клуба)


свойство mode отражает в каком режиме участник клуба пытается манипулировать ресурсом.
если ресурс (энтити) хранит ссылку (не значение, а поле) на сущность участника клуба, то участник имеет отношение прямое к ресурсу.
например ресурс workout имеет свойства sportsman_id, coach_id. если id аутентифицированного участника клуба 
совпадает с одним из свойств sportsman_id, coach_id, то участник имеет непосредственное отношение к ресурсу (self mode)
в противном случае (other mode).
доступ на действия: создания, редактировани или чтения определяется в зависимости от роли, под которой участник просит доступ к русерсу

иные таблицы определяющие доступ: person_role (карта ролей которыми участник обладает) и role (описание ролей системы)
person_role:
- id
- person_id
- role

role:
- role
- description


другие сущности приложения: person (участники клуба) и workout (тренеровка)

person:
- id
- first_name
- last_name
- email
- email_verified
- email_2fa_enabled
- password
- is_active
- crated_at

workout:
- id
- description
- date
- activity
- result
- sportsman_id
- coach_id

прошу изменения за отсутствие описания не во всех сущностях. (как говорил мой преводавтель по физике твердого тела в универа, после молчаливого отображения на доске сложной абстрактной схемы: "из рисунка видно":)

приложение разделено на 3 уровня storage, engine, api c целью возможности потенциальной замены api или storage.

отладка при запуске:
1) применить скрипт postgres_script.sql
2) установить зависимости в .venv из списка requirements.txt  (psycopg[binary], bcrypt, PyJWT)
3) настроить FITNESS_DB_CONNECTION_CREDS опцию подключения к базе данных в settings.py

К сожалению в приложении есть места для улучшений. не все действия реализованы но для тестового
задания, где основная задача реализация системы доступа к ресурсам может быть и достаточно.
